# Paper Formalization Automation — Design

**Date:** 2026-04-22
**Repo:** `git@github.com:liaoruoxue/gaia-papers.git` (`/Users/risoliao/Code/gaia-lb/papers`)
**Runtime:** `/loop 10m` in a long-lived Claude Code session
**Owner:** risoliao

## Goal

Every 10 minutes, pull the repo; if new papers appear in `inbox/`, formalize them into
Gaia packages under `formalized/`, commit + push + tag, one paper per tick.

## Scope

In scope:
- Triggering and structuring the 10-minute loop.
- Detecting new source files in `inbox/`.
- Invoking the `/gaia:formalization` skill fully autonomously (no interactive disambiguation).
- Running `gaia compile` and `gaia check`; pushing and tagging on green.
- Moving source files between `inbox/` and the formalized package.
- Logging failures locally and restoring inbox state so the next tick can retry.

Out of scope:
- Cross-package joint inference (`gaia infer --depth N`).
- Publishing with `/gaia:publish` or rendering with `gaia render`.
- Running any mechanism other than `/loop` (no system cron, no remote scheduled agent).
- Quality review of formalized output (treat as a reviewable first draft).

## Runtime & Cadence

- Mechanism: `/loop 10m "<workflow prompt>"` in the user's Claude Code session.
- Exactly one tick fires every 10 minutes; if a tick is still running when the next
  fire time arrives, the harness queues the next fire. No overlap.
- The loop terminates when the user runs `/loop` stop or closes the session. There is
  no auto-stop condition; empty ticks are fine.

## Per-Tick Workflow

1. `cd /Users/risoliao/Code/gaia-lb/papers && git pull --rebase --autostash`.
   On conflict, abort the tick and log; retry next tick (no auto-resolve).

2. List `inbox/` for files matching `*.pdf`, `*.md`, `*.txt`, `*.html`.
   - Ignore hidden files (`.gitkeep`, `.DS_Store`, etc.).
   - If the list is empty, exit the tick.

3. Sort the list alphabetically (deterministic). Choose the **first** file only —
   exactly one paper per tick. Multiple pending papers drain across consecutive
   ticks.

4. Derive slug from filename: lowercase, replace runs of non-alphanumeric chars
   with `-`, trim leading/trailing `-`, drop extension.
   Example: `Attention Is All You Need.pdf` → `attention-is-all-you-need`.

5. Collision guard: if `formalized/<slug>-gaia/` already exists, skip this paper,
   append a line to `_failed/<slug>-collision.log` (gitignored), and end the tick.
   Operator must delete the existing package to force re-formalization.

6. Scaffold: `cd formalized && gaia init <slug>-gaia`.

7. Move source into the package: `mv inbox/<file> formalized/<slug>-gaia/artifacts/`.

8. Invoke `/gaia:formalization` against the artifact, fully autonomous. The loop
   prompt (see Appendix A) instructs the model to skip every clarifying question
   the skill would normally ask — it commits to best-guess answers and proceeds.
   The model MUST NOT pause for operator input at any point in the tick.

9. Run `gaia compile .` and `gaia check .` inside the package.

10. **On success:**
    - `git add formalized/<slug>-gaia/`
    - `git commit -m "<commit message below>"`
    - `git tag formalized/<slug>`
    - `git push origin main`
    - `git push origin formalized/<slug>`

11. **On failure (any step 8–10 before the tag):**
    - Move source back: `mv formalized/<slug>-gaia/artifacts/<file> inbox/<file>`.
    - Delete the half-baked package: `rm -rf formalized/<slug>-gaia/`.
    - Write a local-only log at `_failed/<slug>-<UTC-timestamp>.log` (path
      gitignored) with stderr + phase of failure.
    - End the tick; next tick will re-attempt the same paper.

## Detection Rule

A file in `inbox/` is "new" iff its derived slug has no matching
`formalized/<slug>-gaia/` directory. Since success moves the source out of
`inbox/` and failure restores it, the inbox is always the authoritative list of
pending work.

## Supported Input Types

`*.pdf`, `*.md`, `*.txt`, `*.html`. `.txt` and `.html` files may contain a URL
(e.g., an arXiv link); the formalization skill is responsible for fetching if
needed. No other extensions are processed.

## Commit Message Template

```
Formalize: <Original Paper Title>

Autogenerated by the 10-min formalization loop.
Source: <original filename>
Package: formalized/<slug>-gaia/
```

`<Original Paper Title>` is derived from the filename stem with capitalization
preserved. No AI attribution footer — this is machine-routine output.

## Tag Format

`formalized/<slug>` on the commit that lands the package. One tag per paper.
Re-formalization (after operator deletes the package) overwrites the tag only
if the operator also deletes the tag — otherwise push fails and falls into the
failure branch.

## Error Handling Summary

| Failure point                      | Behavior                                                                                      |
|------------------------------------|-----------------------------------------------------------------------------------------------|
| `git pull` conflict                | Abort tick, log, retry next tick. No auto-resolve.                                            |
| Slug collision                     | Skip paper, write `_collision.log`, end tick.                                                 |
| Formalization / compile / check    | Move source back to `inbox/`, delete package dir, write `_failed/...log`, end tick.           |
| Tag already exists                 | Treated as a push failure — falls into the failure branch above. Operator must clear the tag. |
| Push fail (network, perms)         | Keep commit local; next tick's `git pull --rebase` rebases onto origin and re-pushes.         |

## Gitignore Additions

Append to `papers/.gitignore`:
```
_failed/
```

## Success Criteria

- Operator drops a PDF into `inbox/`. Within ~10 minutes, a commit and a
  `formalized/<slug>` tag appear on `origin/main`, and the PDF has moved into
  `formalized/<slug>-gaia/artifacts/`.
- Operator drops a PDF that fails `gaia check`. Within one tick, the PDF is
  back in `inbox/`, no package dir exists in `formalized/`, and a log file
  under `_failed/` explains the failure.
- Multiple PDFs dropped at once drain sequentially, one per tick.

## Risks and Mitigations

- **Quality drift:** fully autonomous formalization produces draft-quality
  packages. Mitigation: treat published packages as reviewable drafts;
  operator can delete + re-run manually via `/gaia:formalization` for a
  hand-guided pass.
- **Session death:** `/loop` stops when the Claude Code session closes.
  Mitigation: operator re-issues `/loop 10m ...` to resume. Future work:
  migrate to a scheduled remote agent or local launchd.
- **Repeated failures on same paper:** a persistently-broken paper will retry
  every tick indefinitely. Mitigation: operator inspects `_failed/` and either
  fixes the source or removes the file from `inbox/`.
- **Prompt drift:** the loop prompt is the source of truth for the per-tick
  workflow. Mitigation: keep the prompt short and link to this spec.
